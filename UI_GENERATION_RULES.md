# 产品 PRD 业务页面生成与迭代规范

## 目的

本规范是根据产品 PRD 生成、修改和验收业务页面的统一规则。

保证页面在多次需求迭代后，视觉结构与交互逻辑保持一致，避免每次生成或修改都形成新的页面风格与布局。

## 适用范围

- 根据 [PRD.md](./PRD.md) 新生成业务页面；
- 为既有业务页面增加字段、状态、操作或流程；
- 将低保真原型实现为前端页面；
- 对业务页面进行视觉或信息架构改版。

不适用于纯技术页面、一次性调试页或第三方系统页面；但只要进入产品导航并服务业务用户，即视为业务页面，必须遵守本规范。

## 依据优先级

页面相关工作必须按以下顺序执行：

1. **低保真原型**：页面结构、信息层级、区块位置、主操作路径与视觉布局的唯一基线。
2. **[PRD.md](./PRD.md)**：功能范围、字段、状态、交互规则、异常处理与验收标准。
3. 现有实现：仅作为可复用的技术基础，不得因为已有组件或代码结构而改变原型布局。

当低保真原型与 PRD 存在冲突时，应暂停实现并先更新原型或 PRD，使二者一致；不得自行选择另一套布局。

## 从 PRD 到业务页面的生成流程

1. 从 PRD 提取该页面的用户目标、字段、状态、操作、异常处理和验收条件。
2. 建立或更新低保真原型，确认页面区块、信息层级、字段分组与主操作路径。
3. 以低保真原型为结构基线实现页面；PRD 中新增的功能必须落入原型的既定结构。
4. 如 PRD 新需求需要改变结构，先更新低保真原型并确认，再修改页面代码。
5. 完成后依据 PRD 验收功能与交互，依据低保真验收布局与层级。

## 强制原则

- 低保真原型是页面结构的唯一基线。实现时必须优先还原其区块、顺序、层级与主操作路径。
- PRD 定义页面应具备什么功能和交互，不替代低保真对页面布局的定义。
- 修改既有页面前，先将其与对应低保真原型对齐；不得在旧页面布局上持续堆叠新字段或操作，导致结构漂移。
- 新增需求必须在现有低保真原型上增量迭代。需要新增区块、改变信息层级或改变主路径时，先更新低保真，再开始编码。
- 不得为同一页面另起一套视觉风格、信息架构或操作流程。
- 组件库仅用于实现，不得反向决定页面布局。若现有组件不适合原型，应调整组件组合或样式。
- 页面实现完成后，必须按低保真逐项核对：页面区块、字段分组、主/次操作、状态反馈、空状态和响应式行为。

## 交付检查清单

每次页面新增或改版的交付说明应明确：

- 对应的低保真原型及其版本；
- 本次遵循的 PRD 章节；
- 原型中已实现的区块；
- 因技术或产品决策产生的差异（如有），以及差异是否已同步回原型和 PRD；
- 已完成的视觉与交互验证。

## 当前适用页面

- 模型配置页：以已确认的“模型配置”低保真原型为唯一布局基线。
- 后续页面：须先建立并确认对应低保真原型，才进入页面实现阶段。
